# 修复总结输出模板

严格区分已修改、未修改和已验证、未验证。不得把建议测试写成已通过，也不得在没有证据时声称“无副作用”。

## 代码缺陷修复总结

**修复概要**

- 输入来源：{扫描报告 / 报错日志+代码 / Code Review 意见+代码}
- 修复总数：{N} 条
- 修复成功：{N} 条
- 根因待定（未修改）：{N} 条
- 涉及文件：{文件列表}
- 工作区保护：{Git 状态 / 非 Git，回滚保证说明}

### FIX-001（对应 DEF-001）

- **位置**：{path}:{start_line}-{end_line} / {symbol}
- **根因状态**：{confirmed / pending}
- **出错原因**：{引用当前源码和根因证据；允许使用 other}
- **改动思路**：{最小修改及其原理}
- **前后代码对比**：
  ```diff
  - {原代码，path:line}
  + {改后代码，path:line}
  ```
- **副作用与未验证风险**：
  - 已检查：{调用方、返回类型、异常路径等}
  - 未验证：{运行时、外部接口、并发或环境风险；没有则写“无”}
- **实际验证**：
  - `{命令}`：{passed / failed / skipped} - {真实摘要}

### 根因待定（未修改）

#### DEF-XXX

- **位置**：{path}:{line} / {symbol}
- **问题**：{报告或日志中的原始问题}
- **无法确认原因**：{缺少的契约、调用方、输入或运行时证据}
- **需要补充**：{材料列表}

## 需人工复查

- {核心流程、公共 API、并发、迁移或未执行测试}

## 免责说明

本报告只反映实际提供的材料和实际执行的验证。未验证的运行时行为仍需在目标环境确认；修复后应再次运行 `code-defect-scanner` 复检。
