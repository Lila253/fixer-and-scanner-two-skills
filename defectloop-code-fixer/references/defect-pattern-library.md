# 缺陷模式库（显性 + 隐形）

> 用途：代码缺陷扫描器扫描代码时查表（"代码特征"列）；代码缺陷修复器分析根因时查表（"根因分析"列）。两个技能共用本文件。
> 使用方式：扫描时逐类比对"代码特征"，命中只登记为候选；完成控制流、数据流或契约验证后再登记为缺陷。修复时查"根因分析"构造根因假设。
> 重要：本表给出的是"统计上的常见模式"。扫描发现的每个缺陷必须有代码证据支撑，不得仅凭模式匹配下结论。

## 一、显性缺陷（有明确报错或必然出错）

### 1. 空指针 / None 解引用

- **类别**：显性
- **典型报错**：`NullPointerException`、`AttributeError: 'NoneType' object has no attribute`、`Cannot read property of null`、`panic: runtime error: invalid memory address`
- **代码特征（扫描找什么）**：
  - 对返回值/字段未做空判断即访问属性或方法
  - 链式调用中途可能为 None（如 `a.b.c` 中 b 可能为 null）
  - 集合元素取出后未判空直接用（如 `map.get(key)` 返回 null 后直接用）
  - 对象初始化顺序问题：使用早于赋值
- **根因分析（为什么会错）**：上游函数返回 None 未处理、字典/Map key 不存在、外部依赖返回空、初始化顺序错误。
- **验证方法**：回溯该变量赋值路径，确认是否存在未初始化/赋值为空的分支。

### 2. 类型与转换错误

- **类别**：显性
- **典型报错**：`TypeError`、`ClassCastException`、`ValueError: invalid literal for int()`、`cannot unmarshal`
- **代码特征（扫描找什么）**：
  - 隐式类型转换（字符串与数字混用）
  - JSON 反序列化未校验类型直接使用
  - 函数参数类型与实际传入不符
  - 泛型/接口强转但实际类型不符
- **根因分析（为什么会错）**：上游接口字段类型变更未同步、用户输入未校验、序列化/反序列化类型不匹配。
- **验证方法**：核对输入数据的实际类型与代码期望类型；检查接口契约是否与调用方一致。

### 3. 边界与越界

- **类别**：显性
- **典型报错**：`IndexError: list index out of range`、`ArrayIndexOutOfBoundsException`、`slice bounds out of range`
- **代码特征（扫描找什么）**：
  - 直接用下标访问未做长度判断
  - 循环边界用 `<=` 而非 `<`（off-by-one）
  - 访问 `[length]` 而非 `[length-1]`
  - 空集合/空数组未判空就取元素
  - 动态计算的索引未校验范围
  - 分页/切片计算错误
- **根因分析（为什么会错）**：空集合访问首元素、off-by-one、分页 size 为 0 或超限。
- **验证方法**：代入边界值（0、-1、length、length+1）推演；确认循环起止。

### 4. 资源泄漏与耗尽

- **类别**：显性
- **典型报错**：`Too many open files`、`OutOfMemoryError`、`connection pool exhausted`、`OOM`
- **代码特征（扫描找什么）**：
  - 打开文件/连接/流未关闭
  - 异常路径未释放资源（未用 try-finally / try-with-resources / with 语句）
  - 缓存无上限只增不减
  - 大对象未释放
- **根因分析（为什么会错）**：异常分支未关闭资源、缓存无淘汰策略、连接池配置过小。
- **验证方法**：沿资源生命周期检查是否有释放路径；确认异常路径是否也会释放。

### 5. 外部依赖与集成

- **类别**：显性
- **典型报错**：`Connection refused`、`timeout`、`5xx`、`ECONNRESET`、第三方接口返回非预期格式
- **代码特征（扫描找什么）**：
  - 未处理超时/重试的外部调用
  - 未校验第三方返回结构
  - 硬编码 URL/密钥
  - 无降级策略
- **根因分析（为什么会错）**：第三方接口变更、网络抖动未重试、未做降级。
- **验证方法**：核对第三方接口契约与本地解析逻辑；确认超时与重试策略。

### 6. 配置与环境

- **类别**：显性
- **典型报错**：`KeyError`/配置缺失、`FileNotFoundException`、环境变量未生效、仅在特定环境复现
- **代码特征（扫描找什么）**：
  - 读取配置未做默认值兜底
  - 环境间配置不一致
  - 硬编码路径
  - 环境变量名拼错
- **根因分析（为什么会错）**：配置项缺失、环境变量名拼错、多环境配置不同步。
- **验证方法**：对比各环境配置项；确认配置加载顺序与覆盖关系。

## 二、隐形缺陷（无报错、静默失败、功能缺失）

### 7. 吞异常 / 静默失败

- **类别**：隐形
- **症状表现**：操作后预期效果没出现；某状态没变化；该发的通知没发；该扣的库存没扣。
- **代码特征（扫描找什么）**：
  - `catch`/`except` 块为空、或只有注释
  - 捕获异常后既不抛出也不记录，静默跳过核心逻辑
  - `catch` 后返回 null/空集合/0/默认值，上游当作正常结果使用
  - `except: pass`（Python）、`catch (Exception e) {}`（Java）
- **根因分析（为什么会错）**：开发者意图"容错"，但吞掉了不该吞的异常，导致核心逻辑被跳过而调用方无感知。
- **验证方法**：在 catch 块加日志确认是否被触发；检查 catch 后的返回值是否被上游当作正常结果。

### 8. 分支遗漏

- **类别**：隐形
- **症状表现**：某类输入落到空分支什么都不做；特定条件下功能不生效。
- **代码特征（扫描找什么）**：
  - `if-elif` 没有 `else` 兜底且某类输入无处理
  - `switch` 缺少 `default`
  - 只处理了部分情况，遗漏了某类输入
- **根因分析（为什么会错）**：需求理解不全，遗漏了某类业务场景的处理分支。
- **验证方法**：枚举所有可能的输入类型，逐一验证是否有对应分支处理。

### 9. 状态机漏状态

- **类别**：隐形
- **症状表现**：状态卡在原状态不变；流程走到某一步就停了。
- **代码特征（扫描找什么）**：
  - 状态转换函数缺少某个状态的转换处理
  - 状态机迁移遗漏分支
  - 某个状态转换只在异步路径处理，同步路径漏了
- **根因分析（为什么会错）**：状态机设计时遗漏了某个状态或某条转换路径。
- **验证方法**：画出完整状态机图，检查所有状态的转换是否都有处理；确认同步/异步路径是否都覆盖。

### 10. 静默 return

- **类别**：隐形
- **症状表现**：功能没生效但无报错；调用方不知道被跳过。
- **代码特征（扫描找什么）**：
  - 中间校验 `return` 了但无日志
  - 某个条件不满足时直接 return，调用方不知道被跳过
  - 提前 return 导致后续主逻辑未执行
- **根因分析（为什么会错）**：防御性 return 没有通知调用方，导致调用方以为执行成功。
- **验证方法**：在 return 前加日志确认是否被触发；检查调用方是否处理了"被跳过"的情况。

### 11. 无超时等待 / 卡死

- **类别**：隐形
- **症状表现**：接口一直不返回；页面转圈；线程堆积；最终超时或手动重启才恢复。
- **代码特征（扫描找什么）**：
  - 等 DB/HTTP/Redis 响应但无超时设置
  - 锁等待无超时
  - `while` 循环条件可能永远为真
  - 递归终止条件可能不可达
  - 获取锁后异常路径未释放（漏 unlock）
  - 连接池/线程池打满后无限等待
- **根因分析（为什么会错）**：外部依赖可能卡住、锁可能不释放、循环可能不终止，但代码无超时兜底导致整体卡死。
- **验证方法**：抓线程 dump 看卡在哪个栈帧；检查所有外部调用是否有超时设置；检查锁是否在 try-finally 中。

### 12. 并发竞态

- **类别**：显性 + 隐形（可报错也可静默不一致）
- **典型报错**：`ConcurrentModificationException`、`fatal error: concurrent map writes`、死锁
- **症状表现**（隐形时）：数据偶发不一致；并发下后写覆盖先写。
- **代码特征（扫描找什么）**：
  - 共享变量无锁/无原子保护
  - 读写同一资源无同步
  - 锁顺序不一致（可能死锁）
  - check-then-act 存在时间窗口
  - 遍历集合的同时修改集合
- **根因分析（为什么会错）**：遗漏同步、双重检查锁写错、缓存与 DB 双写不一致。
- **验证方法**：确认是否有共享可变状态；确认保护机制是否覆盖所有访问路径；检查锁顺序。

### 13. N+1 查询 / 性能退化

- **类别**：隐形
- **症状表现**：接口响应时间逐渐变长；偶发性变慢；重启后恢复一段时间又变慢。
- **代码特征（扫描找什么）**：
  - 循环内单条 DB/远程调用查询
  - 全局集合只增不减（缓存无淘汰、列表无清理）
  - SQL 无索引/索引失效
  - 每次请求都全量计算本可缓存的结果
  - 频繁创建大对象导致 GC 压力
- **根因分析（为什么会错）**：列表越大查询越多；资源只增不减；无索引导致全表扫描。
- **验证方法**：慢查询日志定位 SQL；检查循环内是否有 DB/远程调用；监控内存/连接数是否持续上涨。

### 14. 逻辑写反

- **类别**：隐形
- **症状表现**：结果不符合预期但无报错；逻辑走错分支。
- **代码特征（扫描找什么）**：
  - `if` 判断条件写反（如 `==` 写成 `!=`、`>` 写成 `<`）
  - 默认值错误
  - 条件判断与业务意图相反
- **根因分析（为什么会错）**：需求理解偏差、编码笔误、边界判断写反。
- **验证方法**：将代码实际逻辑与需求描述的预期行为逐条比对，定位差异点。

## 使用提示

- 一次扫描可能命中多个模式（如并发引发空指针），需分别给出代码证据。
- 扫描时以"代码特征"列为候选线索，逐个文件逐个函数比对；完成证据验证后再登记为缺陷。
- 修复时以"根因分析"列为参考，构造根因假设，但最终根因必须由代码证据支撑。
- 若代码特征不匹配任何模式，不要强行归类，标注"待确认"。
- 允许使用"其他"类别；不要为了套用模式库而修改问题描述或根因。
- 本模式库为辅助工具，不得仅凭模式匹配下结论，必须有代码证据。
