# 报错类型 → 代码模式映射表

> 用途：当用户附带报错日志作为线索时，根据报错类型优先扫描对应代码模式。
> 使用方式：在堆栈中识别报错类型 → 查下表 → 优先扫描"可能代码模式"对应的代码位置。
> 重要：报错作为线索时用于聚焦扫描范围，不是必须输入。核心扫描依据是 defect-pattern-library.md。

## 1. 空指针 / None 解引用类

**典型报错**：`NullPointerException`(Java)、`AttributeError: 'NoneType' object has no attribute`(Python)、`nil pointer dereference`(Go)、`NullReferenceException`(C#)、`Cannot read property of undefined/null`(JS)

**可能代码模式**：
- 对外部输入/接口返回值未判空，直接访问属性或方法
- 链式调用中间环节可能为空（如 `a.b.c` 中 b 为 null）
- 集合元素取出后未判空（如 `map.get(key)` 返回 null 后直接用）
- 对象初始化顺序问题：使用早于赋值

**优先扫描位置**：
- 堆栈指向的函数中所有"外部数据来源"（DB查询、接口返回、配置读取、用户输入）的取值点
- 链式调用中每一环是否有判空
- 该对象在所有代码路径上是否都被正确赋值

## 2. 索引越界类

**典型报错**：`IndexError: list index out of range`(Python)、`ArrayIndexOutOfBoundsException`(Java)、`IndexOutOfRange`(C#)

**可能代码模式**：
- 循环边界用 `<=` 而非 `<`（off-by-one）
- 访问 `[length]` 而非 `[length-1]`
- 空集合/空数组未判空就直接取元素
- 动态计算的索引未校验范围

**优先扫描位置**：
- 堆栈指向的取下标操作，核对索引上界
- 循环条件与集合实际长度的关系
- 集合是否可能为空

## 3. 键缺失类

**典型报错**：`KeyError`(Python)、`NoSuchElementException`(Java)、`undefined`(JS 访问对象属性)

**可能代码模式**：
- 直接用 `[key]` 访问字典/Map，未用 `get(key, default)` 或 `containsKey` 预检
- 配置项缺失但代码假设它一定存在
- 不同环境（开发/生产）配置不一致导致键缺失

**优先扫描位置**：
- 堆栈指向的取键操作，确认 key 的来源
- 该 key 在所有环境下是否都会被写入
- 是否有默认值兜底

## 4. 类型错误类

**典型报错**：`TypeError`(Python/JS)、`ClassCastException`(Java)、`InvalidCastException`(C#)

**可能代码模式**：
- 函数参数类型与实际传入不符
- 反序列化后未校验类型直接使用
- 隐式类型转换出错（如把 None 当数字参与运算）
- 泛型/接口强转但实际类型不符

**优先扫描位置**：
- 堆栈指向的操作，核对操作数的实际类型
- 追溯数据来源，看类型在哪一步被改变或未校验
- 检查反序列化逻辑是否做了类型断言

## 5. 超时类

**典型报错**：`TimeoutException`、`ConnectionTimeout`、`ReadTimeout`、`socket timeout`

**可能代码模式**：
- 死循环 / 无终止条件的递归
- 资源竞争导致阻塞（锁未释放、连接池耗尽）
- 外部依赖（DB/HTTP/Redis）慢查询或不可达，无超时兜底
- 重试风暴：重试间无退避，放大延迟

**优先扫描位置**：
- 确认超时发生在哪一层（网络/DB/计算）
- 是否有无界循环或无超时的外部调用
- 资源占用（连接池、线程池、锁）是否打满

---

## 6. 连接 / 网络类

**典型报错**：`ConnectionRefused`、`ConnectionReset`、`SocketException`、`ECONNREFUSED`

**可能代码模式**：
- 目标服务未启动或端口不对
- 防火墙/安全组拦截
- 连接未复用、未关闭导致句柄耗尽后新建连接失败
- DNS 解析失败

**优先扫描位置**：
- 确认目标地址/端口是否可达
- 连接池配置与实际连接数
- 是否有连接泄漏（打开未关闭）

---

## 7. 内存类

**典型报错**：`OutOfMemoryError`(Java)、`MemoryError`(Python)、`JavaScript heap out of memory`

**可能代码模式**：
- 大对象/大集合一次性加载（如全表查询）
- 资源未释放（文件句柄、连接、游标）
- 内存泄漏：持续累积但从不清理的缓存/列表
- 递归无终止导致栈/堆爆

**优先扫描位置**：
- 堆栈是否指向大对象分配点
- 是否有全局可变集合只增不减
- 资源（流/连接）是否在 finally 或 with 中释放

---

## 8. 数据库类

**典型报错**：`SQLException`、`IntegrityError`、`DeadlockDetected`、`OperationalError`

**可能代码模式**：
- SQL 拼接导致语法错误或注入
- 唯一约束/外键约束冲突（重复插入、引用不存在的记录）
- 死锁：多个事务以不同顺序锁同一批表
- 连接泄漏：事务未提交/回滚，连接未归还

**优先扫描位置**：
- 具体SQL 错误码，定位是约束/死锁/语法
- 事务边界与锁顺序
- 连接是否在异常路径上也归还了

---

## 9. 并发类

**典型报错**：`ConcurrentModificationException`(Java)、`Deadlock`、`RaceCondition`（通常表现为数据不一致而非报错）

**可能代码模式**：
- 遍历集合的同时修改集合
- 共享变量未加锁/未用原子操作
- 锁顺序不一致导致死锁
- 检查-执行（check-then-act）存在时间窗口

**优先扫描位置**：
- 堆栈是否涉及多线程/多协程
- 共享状态的同步机制
- 锁的获取顺序是否全局一致

---

## 10. 栈溢出 / 递归类

**典型报错**：`StackOverflowError`(Java)、`RecursionError`(Python)、`Maximum call stack size exceeded`(JS)

**可能代码模式**：
- 递归缺少终止条件
- 终止条件永远不满足（如边界判断写反）
- 相互递归形成环
- 递归深度随输入线性增长且无尾递归优化

**优先扫描位置**：
- 堆栈是否大量重复同一函数（递归未终止的标志）
- 递归终止条件是否可达
- 评估输入规模是否会触发过深递归

## 使用提示

- 报错类型不在上表时，按"报错关键词 + 触发场景"自行归纳可能代码模式，并明确标注"推断"。
- 一个报错可能匹配多个模式，全部列出并按可能性排序。
- 报错作为线索时用于聚焦扫描范围，找到后仍需在 defect-pattern-library.md 中匹配具体缺陷类型并给出代码证据。
