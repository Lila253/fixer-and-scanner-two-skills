# 严重度分级标准（P0-P3）

本文件用于代码缺陷扫描器的严重度分级步骤。严重度表示问题一旦成立时的影响，不表示证据强弱；置信度和状态由扫描器另行记录。

## P0 — 致命（立即阻断）

定义：导致系统不可用、核心功能完全中断、数据丢失或损坏、存在安全漏洞可被利用。

判定条件（满足其一即可）：
- 线上服务整体不可用或核心接口 100% 失败。
- 产生脏数据、数据丢失、资金/订单错误。
- 可被远程利用的安全漏洞（注入、越权、敏感信息泄露）。

处置要求：必须立即修复并回滚/热修；缺陷报告中标注"阻断发布"。

## P1 — 严重（影响主流程）

定义：核心功能部分受损、高频路径出错、有可用绕过但体验严重下降。

判定条件（满足其一即可）：
- 核心功能在特定条件下报错或返回错误结果。
- 影响面较大但未完全中断。
- 有临时绕过方案，但用户感知明显。

处置要求：当版本必须修复；缺陷报告中给出临时绕过措施。

## P2 — 一般（影响次要路径）

定义：非核心功能异常、边界场景出错、影响面有限。

判定条件（满足其一即可）：
- 次要功能或低频路径报错。
- 仅在极端输入或特定环境下复现。
- 体验受损但不阻断主流程。

处置要求：纳入下个迭代修复；缺陷报告应可排期。

## P3 — 轻微（优化项）

定义：不影响功能正确性，且属于低风险性能优化或维护性改进。纯命名、格式、文档和注释建议默认标为 `kind=suggestion`，不计入缺陷总数，不强行分配 P3。

判定条件（满足其一即可）：
- 代码坏味道、重复逻辑、命名不规范。
- 非关键性能瓶颈。
- 文档/注释缺失、日志冗余。

处置要求：择机优化；不阻塞发布。

## 定级输出格式

对每条缺陷，输出：
```
严重度：P{级别}
定级理由：{一句话说明为何定此级别，引用具体影响}
```

## 置信度与严重度的关系

- 置信度反映"缺陷判断的可信程度"，严重度反映"问题本身的影响程度"，二者独立。
- 扫描发现的缺陷，若代码证据充分（直接引用了问题代码行），置信度可标"高"。
- 若代码证据间接（如调用链推断可能有问题），置信度标"中"，并注明"需进一步确认"。
- 置信度低的缺陷可以列出，但必须与高置信度缺陷分开排序，标注"待确认"。
- 严重度高（P0）但置信度低的缺陷，修复时应优先验证而非直接修复。

## 定级前检查

- 先写出影响假设：受影响的功能、可达条件、数据范围和是否存在绕过方案。
- 没有业务影响或可达性证据时，不得判为 P0；安全 P0 还必须说明攻击面或敏感数据影响。
- `kind=suggestion` 不得伪装成缺陷来增加缺陷数量。
- `status=hypothesis` 的问题可以列出，但只能进入待确认区，不能直接交给修复器修改。
