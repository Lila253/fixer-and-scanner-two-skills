# 代码缺陷扫描器

## 职责边界

检查用户授权范围内的代码，结合现有项目工具与语义分析生成有证据的缺陷报告。把模式库当作搜索线索，不把模式命中直接当作缺陷结论。

保持只读：不得修改源码、配置、测试或依赖文件。用户同时要求“检查并修复”时，先完成扫描并生成报告，再把已确认缺陷交给 `code-defect-fixer`；扫描阶段仍不得改代码。

## 意图分流

- 用户要求“检查、扫描、自检、审查缺陷、定位报错、验证修复”时执行本流程。
- 用户只要求解释现有报告时，解释报告，不重新扫描，除非用户要求复检。
- 用户明确要求修改代码且不要求先扫描时，交给 `code-defect-fixer`。
- 用户只要求格式化、重命名、重构、架构建议、需求分析或文档编写时，不使用本 skill。
- 用户只提供报错但没有代码、堆栈路径或项目范围时，先索取可定位的代码来源，不猜测文件位置。

## 输入与默认值

接受以下代码来源之一：项目目录、Git 改动、指定目录、指定文件或函数、粘贴代码片段。

- 优先采用用户明确指定的范围。
- “提交前检查”或“检查 PR 改动”默认读取 Git diff；若当前位置不是 Git 仓库，说明原因并请用户提供范围，不假装完成 diff 扫描。
- 用户要求“检查当前项目”且当前工作目录明确时，直接采用当前项目，不重复询问。
- 对粘贴片段使用 `pasted-snippet` 作为路径，行号从片段第一行开始；明确说明无法分析未提供的调用方。
- 把报错日志、症状和用户怀疑方向作为聚焦线索，不把它们当作缺陷证据。

## 工作流程

### 1. 确定范围并建立清单

优先使用 `rg --files` 或 Git 文件列表，遵守 `.gitignore`。默认排除 `.git`、`node_modules`、`vendor`、`dist`、`build`、`coverage`、缓存目录、压缩包、二进制、压缩/生成代码和锁文件；用户明确要求时再纳入。

跳过单个超过 1 MiB 的文本文件并记录原因。不得读取 `.env`、私钥、凭据文件或范围外路径；若用户明确要求检查敏感文件，只引用必要片段并遮盖密钥值。

记录已扫描、已跳过和无法读取的文件。这里的覆盖率只表示“扫描范围文件覆盖”，不得冒充运行时测试覆盖率或路径覆盖率。

### 2. 检测项目工具

从项目配置识别语言、构建系统和已有检查命令。优先使用仓库已经配置的编译、测试、lint、类型检查和静态分析工具；不得为了扫描自行安装依赖或工具。

对每个命令记录 `passed`、`failed` 或 `skipped` 以及摘要。工具失败是线索，不自动等同于代码缺陷；先定位到用户范围内的具体代码证据。

### 3. 聚焦代码路径

- 有堆栈时，先检查堆栈指向的函数及直接上下游。
- 有 Git diff 时，先检查改动行，再检查受影响的调用方、异常路径和资源生命周期。
- 扫描指定函数时，只扩展到证明或否定候选问题所必需的调用方。
- 扫描整个项目时，分语言和模块处理；无法完整建立调用链时如实标记部分覆盖。

### 4. 识别候选并验证

按需读取 `references/defect-pattern-library.md`。用户提供错误日志时，再读取 `references/error-type-pattern-map.md`。模式仅用于发现候选位置；登记前必须检查控制流、数据来源、接口契约和相关调用方。

允许使用模式库之外的 `other` 类别。不得为了套用分类而歪曲根因。

为每个发现设置种类与状态：

- `kind=defect`：存在可导致错误行为的实现问题。
- `kind=risk`：存在现实风险，但触发条件仍需外部契约或运行时信息确认。
- `kind=suggestion`：不影响当前正确性的改进项，不计入缺陷总数。
- `status=confirmed`：测试、编译器、静态工具或完整控制流证明问题成立。
- `status=probable`：局部数据流证据充分，但依赖一个明确写出的外部假设。
- `status=hypothesis`：需要业务规则、接口契约或运行时材料验证；单独列出，不交给修复器自动修改。

### 5. 分级并控制误报

读取 `references/severity-standard.md`。严重度表示影响，置信度表示证据强度，两者独立。

- 没有业务重要性、可达性或影响范围证据时，不得判为 P0。
- 仅有模式匹配时，不得标为 `confirmed`。
- “逻辑写反、分支遗漏、静默 return”等依赖业务意图的候选，在没有需求或调用方证据时标为 `hypothesis`。
- 对无法确认的问题说明缺少什么材料，不用推测补齐事实。

### 6. 输出并校验报告

读取 `references/defect-report-schema.json` 和 `references/defect-report-template.md`。以符合 schema 的 JSON 作为两个 skill 之间的权威输入，同时给出简短 Markdown 摘要供用户阅读。

保持 `DEF-001` 起的连续编号。每条证据包含相对路径、准确行号和代码原文。报告写入文件时运行：

```text
python scripts/validate_report.py <report.json> --source-root <扫描根目录>
```

若校验失败，修正报告后再交付。只有聊天内 JSON、无法落盘时，逐字段人工核对并说明未运行脚本校验。

## 完成条件

仅在以下条件全部满足时声称扫描完成：

- 已说明实际扫描范围、跳过项和文件覆盖情况。
- 每个 defect/risk 都有真实代码证据和影响假设。
- 已区分 confirmed、probable 与 hypothesis。
- 已记录实际运行的工具以及未运行原因。
- 未修改任何用户代码或配置。

## 配套资源

- `references/defect-pattern-library.md`：候选缺陷模式；只在语义扫描阶段读取。
- `references/error-type-pattern-map.md`：错误日志到候选模式的路由；仅在有日志时读取。
- `references/severity-standard.md`：严重度、置信度和状态判定规则。
- `references/defect-report-schema.json`：扫描器与修复器共享的报告协议。
- `references/defect-report-template.md`：用户可读摘要和 JSON 输出示例。
- `scripts/validate_report.py`：无需第三方依赖的报告校验器。
