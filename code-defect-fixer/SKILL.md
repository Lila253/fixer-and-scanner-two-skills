# 代码缺陷修复器

## 职责边界

仅在用户明确要求“修复、改代码、应用报告、落实 Review 意见”等修改动作时写入文件。只修复已经有足够证据的问题；诊断、解释报告、提出方案但不修改的请求，不执行本流程。

优先做最小、可回顾的补丁，不主动重构、换库或扩大范围。缺陷模式库是根因分析参考，不是必须命中的分类表；允许使用 `other`，不得为了套用模式库而制造根因。

## 意图分流

- “解释这个报错/报告”或“给修复方案但不要改代码”时，只分析，不写文件。
- “根据 DEF-001 修复”“根据报错修改”“按 Review 意见改代码”时执行本流程。
- “检查并修复”时，先交给 `code-defect-scanner` 扫描，再只修复确认过的缺陷，最后复检。
- 没有代码、项目目录、报告中的有效路径或用户明确授权的文件时，先索取材料，不猜测路径。
- 涉及纯格式化、纯重构、需求实现或文档修改时，不把请求伪装成缺陷修复。

## 输入要求

接受以下入口之一：

- `code-defect-scanner` 生成的 `schema_version=1.0` JSON 报告，或可被规范化为该 JSON 的报告。
- 报错日志/堆栈与对应代码。
- Code Review 意见与对应代码。
- 用户已明确定位的文件、函数和问题。

用户明确要求修复报告中的问题时，报告列出的路径可作为候选范围；仍需检查当前源码并确认这些路径确实是本次授权范围。仅提供报告但没有修改意图时，不得自动修改。

## 工作流程

### 1. 确认权限和工作区状态

读取用户明确授权的文件，拒绝访问范围外路径。若在 Git 仓库中，先运行 `git status --short` 和必要的 `git diff --check`，保留用户已有改动，不使用 `git reset --hard`、`git checkout --` 或覆盖式写入。

若不在 Git 仓库，说明无法提供 Git 回滚保证，并保留实际修改前的代码片段用于报告 diff；不要创建无关备份文件污染项目。

### 2. 解析并验证输入

对 JSON 报告运行：

```text
python scripts/validate_report.py <report.json> --source-root <项目根目录>
```

校验失败、报告缺少证据或报告中的证据与当前源码不匹配时，先标记报告过期/材料不足，不直接按旧行号修改。Markdown 报告必须先人工规范化为相同字段，并说明未运行机器校验器。

从报错入口提取异常类型、消息、堆栈帧和触发条件；从 Review 入口提取逐条意见。每条登记为 `FIX-*`，并保留原始材料。

### 3. 分析根因

逐条读取当前代码和必要的上下游调用方，使用 `references/defect-pattern-library.md` 的根因列作为参考。必须回答：

- 哪个输入、状态或控制流导致错误。
- 证据来自哪些当前代码行或工具结果。
- 哪些假设仍未验证。
- 是否存在多个叠加根因。

根因无法确认时，进入“根因待定”，列出所需材料，不修改该条。不要仅凭静态模式或旧报告行号修复。

### 4. 设计并确认补丁

采用与仓库现有风格一致的最小改动。以下情况视为大范围变更，执行前先向用户说明文件、行为和验证计划并等待确认：多于一个源码文件、公共 API/数据结构变化、数据库迁移、认证授权、并发同步或核心交易流程变化。

需要新增/修改测试时，必须确认测试文件也在用户授权范围；未获授权时只给出测试建议，不擅自扩展文件范围。

### 5. 执行和验证

使用 `apply_patch` 或等价的最小编辑方式，记录每个文件的原始片段、修改片段和实际 diff。修改后按可用性从近到远执行：目标测试、相关测试、编译/类型检查/lint、`git diff --check`。不得为了通过检查而安装依赖；工具不可用时明确写 `skipped` 和原因。

至少验证以下路径：触发缺陷的边界输入、正常输入、相关异常路径和调用方兼容性。完成后重新读取修改文件，确认报告中的缺陷已消失且没有明显新缺陷。

### 6. 输出结果

读取 `references/fix-summary-template.md`，输出每条 `FIX-*` 的根因、改动思路、真实 diff、已观察到的副作用、未验证风险和可执行测试结果。

副作用字段不得未经证据写“无副作用”。必须区分：已验证、静态检查覆盖、未验证。报告末尾列出成功、失败、根因待定、实际运行的命令和需人工复查点。

## 硬约束

1. 没有明确修改意图，不得写入任何文件。
2. 不修改用户未授权的路径，不覆盖用户已有工作区改动。
3. 根因未确定、报告证据过期或代码位置无法确认时，不强行修复。
4. 不编造文件、函数、变量、行号、测试结果或副作用结论。
5. 多文件或核心逻辑变更必须先确认方案。
6. 完成修复后必须报告实际验证状态；不能把建议测试写成已通过。
7. 修复完成后建议使用 `code-defect-scanner` 复检，但不得声称复检已经执行，除非确实执行。

## 配套资源

- `references/defect-pattern-library.md`：根因分析参考；不强制所有问题匹配其中的类别。
- `references/defect-report-schema.json`：扫描器报告协议副本，用于校验输入。
- `references/fix-summary-template.md`：用户可读的修复总结模板。
- `references/verification-policy.md`：按项目工具和改动风险选择验证命令的规则。
- `scripts/validate_report.py`：无需第三方依赖的报告校验器。
